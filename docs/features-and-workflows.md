# Gastown Features and Workflows

This doc is a combined, source-backed reference for:

- The classification contract used for feature and workflow inventories.
- The current feature inventory and workflow inventory, as extracted into evidence artifacts.

Primary sources:

- Classification contract: `.sisyphus/evidence/task-2-classification-contract.md`
- Feature inventory evidence: `.sisyphus/evidence/task-4-gastown-features.md`
- Workflow inventory evidence: `.sisyphus/evidence/task-5-gastown-workflows.md`

Note on paths: evidence paths are repo-relative. Many point into the `gastown/` subdirectory (for example `gastown/internal/web/api.go`).

## Glossary

- Evidence seed: A `.sisyphus/evidence/...` artifact that records extracted facts plus citations.
- Evidence path: A repo-relative pointer to code, tests, captured output, or evidence seed lines.
- Surface: A user-facing interface where interaction happens, for example a CLI command or HTTP route.
- Feature: A discrete capability exposed at a user-facing surface.
- Workflow: A repeatable end-to-end sequence that uses multiple steps or surfaces to accomplish an outcome.
- Status token: One of `implemented`, `partially_implemented`, `indicated`, `roadmap`, `deprecated`, `unknown/conflict`.
- Confidence: A qualitative strength label for the evidence supporting a claim.
- Cross-repo contract: A shared assumption across repos, often an env var, flag, path, or semantic.
- Conflict register: The place cross-repo contract mismatches are logged, using `unknown/conflict` when unresolved.
- Orchestrator: A process that coordinates another tool or repo, often by spawning it as a subprocess.

## Feature Classification Model

Canonical contract: `.sisyphus/evidence/task-2-classification-contract.md`.

Feature
- A discrete capability exposed at a user-facing surface (CLI command, HTTP route, file format, integration point).

Workflow
- A repeatable, end-to-end sequence that orchestrates multiple steps or surfaces to accomplish a larger outcome.

Rule of thumb
- If you can describe it as one interaction at one surface, it is usually a feature.
- If you need multiple interactions or multiple surfaces to describe it accurately, it is usually a workflow.

Status taxonomy (locked)

Only these status tokens are allowed:

- `implemented`
- `partially_implemented`
- `indicated`
- `roadmap`
- `deprecated`
- `unknown/conflict`

Status definitions (condensed)

`implemented`
- Meaning: User-visible behavior exists now.
- Hard rule (features): requires non-doc evidence.
- Hard rule (workflows): requires at least one non-doc evidence pointer.

`partially_implemented`
- Meaning: Some user-visible behavior exists, but parts are stubbed, no-op, gated, or explicitly not yet implemented.

`indicated`
- Meaning: Strong signals exist that a feature or workflow is intended, but implementation is not verified.

`roadmap`
- Meaning: Future intent with no usable behavior today.

`deprecated`
- Meaning: Feature or workflow exists or existed, but is flagged as deprecated, replaced, hidden, or scheduled for removal.

`unknown/conflict`
- Meaning: Evidence is contradictory or incomplete in a way that cannot be resolved with the available sources.

Confidence levels

Confidence is a separate field from status:

- High
- Medium
- Low

Evidence types and source precedence (highest wins on conflict)

1. Code and handlers (implementation source)
2. Executable surfaces (runtime proof)
3. Tests
4. Docs (reference)
5. Design docs (intent)
6. TODO/FIXME markers (intent signal)

Conflict handling (condensed)

- Prefer the highest-precedence source.
- If two sources at the same precedence disagree and you cannot prove which is correct, mark status as `unknown/conflict` and record both claims with evidence paths.
- If evidence is missing, use `indicated` or `roadmap` (not `implemented`).

Feature inventory row schema (required fields)

- surface: Where the user interacts (CLI command, HTTP route, proxy endpoint, file format, integration tool).
- invocation: Exact example of how it is invoked (command line, request shape, API path).
- inputs: User-controlled inputs and required parameters.
- outputs: User-visible outputs (stdout, JSON, returned data).
- side effects: Persistent changes, network calls, file writes, state mutations.
- evidence path: Concrete pointer(s) to evidence (file paths, test names, or captured command output paths).

Workflow inventory row schema (required fields)

- workflow: Human name for the end-to-end workflow.
- trigger: How the workflow starts (CLI, HTTP request, scheduled job, background worker trigger, external event).
- invocation: Exact example of how it is invoked (command line, request shape, API path, schedule expression).
- steps: Ordered list of major steps (what happens, in what order).
- surfaces touched: The concrete surfaces involved (commands, routes, queues, files) and where each step lives.
- inputs: User-controlled inputs and required parameters.
- outputs: User-visible outputs and completion signals (stdout, return value, emitted artifact, state transition).
- side effects: Persistent changes, network calls, file writes, state mutations.
- evidence path: Concrete pointer(s) to evidence (orchestrator code, step code, test names, captured runtime output paths).
